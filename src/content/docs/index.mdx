Unitflow brings UI behavior into the Effect runtime. Models are UI-facing
Effect Services: they own state, actions, async work, child models, and
lifetime, and they compose through Effect dependency injection.

The core idea: build UI models with the same layers, scopes, streams, fibers,
typed errors, schedules, schemas, and test replacement you already use in
Effect. A View only binds the model's `ui` to the screen.

## Why It Exists

Unitflow does not ask you to adopt a separate UI framework. It adds small
UI-shaped primitives on top of Effect.

<div class="unitflow-cards">
  <div class="unitflow-card">

### Compose via Effect DI

Models are real Effect Services. Layers, Scope, Stream, Schema, and Schedule
are already available inside `make` — no adapter layer to learn.

  </div>
  <div class="unitflow-card">

### Explicit Boundaries

The `inputs` / `outputs` / `ui` contract is enforced by the type system: a
model without `ui` cannot bind to a View.

  </div>

### Agent-Friendly

A small, regular API an LLM can follow reliably, backed by a compact skill
file that teaches the conventions in one read.

  </div>
  <div class="unitflow-card">

### Test Headlessly

`Registry.allSettled` drives full async and reactive flows headlessly — no
mocks, no DOM.

## Install

For React apps, install the React package with its peers:

<Tabs>
  <TabItem label="pnpm">

```sh
pnpm add @unitflow/react @unitflow/core effect@4.0.0-beta.88 react
```

  </TabItem>
  <TabItem label="npm">

```sh
npm install @unitflow/react @unitflow/core effect@4.0.0-beta.88 react
```

  </TabItem>
  <TabItem label="yarn">

```sh
yarn add @unitflow/react @unitflow/core effect@4.0.0-beta.88 react
```

  </TabItem>
</Tabs>

For non-React runtimes, install the core package:

```sh
pnpm add @unitflow/core effect@4.0.0-beta.88
```

## See It in Context

A task board shows model boundaries end to end: each task owns its own state
and events, the board owns the dynamic list and derives aggregate state, and
tests drive the same public actions the UI does — no mocking required.

## Reading Order

Read the docs in the same order Unitflow code is usually built:

```txt
Install           -> add the packages
Stores            -> state: make, get, set, derive, changed
Events            -> actions: make, emit, handler
Model             -> Effect service, contract, dependencies, children
Queries           -> async reads
Mutations         -> async writes
React Binding     -> render the model's ui
Testing           -> layers, fakes, allSettled
Streams           -> advanced pipelines
Lifetime          -> scopes, TTL, finalizers
```

Then apply it to UI through React Views that render model `ui`.
